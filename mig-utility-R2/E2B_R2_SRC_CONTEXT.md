# E2B R2 to SRC Load Context

## Purpose

This file captures how `load_e2b_r2_to_src.py` reads E2B(R2) XML files from `input/e2b_r2_xml` and loads them into the SRC tables defined in `db/ddl/00_create_e2b_r2_src_tables.sql`. Migration and staging control tables (`MG_CASE_LIST`, `MG_CODELIST_MAP`, and related objects) are defined in `db/ddl/01_create_src_tables.sql` instead.

Use this as the working context for the next step: writing `MINUS` SQLs between the SRC tables and the corresponding `ARGUS_APP` case tables to validate migration completeness and correctness.

## What the loader actually does

The Python loader is metadata-driven:

1. It reads the SRC table definitions from `db/ddl/00_create_e2b_r2_src_tables.sql`.
2. For each `CREATE TABLE`, it extracts the column list in DDL order.
3. It parses each XML file under `input/e2b_r2_xml` in sorted filename order.
4. It walks the XML only through the node hierarchy hardcoded in `TABLE_CONTEXT`.
5. For each row:
   - sequence/key columns are generated by the script
   - every other destination column is populated from the XML child element whose tag name equals the lowercase column name
6. If an XML tag is missing, the destination column is inserted as `NULL`.
7. If an XML tag exists but there is no matching column in the DDL, that XML value is ignored.
8. No business transformation is applied. Values are loaded as trimmed text strings.
9. All SRC columns are `VARCHAR2(4000)`, so the loader does not perform type casting.

In short: the loader is mostly a direct XML child-tag to SRC column copy, plus generated sequence keys.

## Important loader rules for reconciliation

- `ICHICSR_SEQ` is not from the XML. It is the 1-based order of the XML file in the sorted input file list.
- `SAFETYREPORT_SEQ` is not from the XML. It is the 1-based order of `<safetyreport>` within a file.
- Child sequence columns such as `PRIMARYSOURCE_SEQ`, `REACTION_SEQ`, `DRUG_SEQ`, `ACTIVESUBSTANCE_SEQ`, etc. are also generated as 1-based counters within their immediate parent node.
- The XML message header (`/ichicsr/ichicsrmessageheader`) is not loaded into any SRC table.
- The loader only visits the XML nodes listed in `TABLE_CONTEXT`. Any XML section outside that hierarchy is ignored.
- Parent tables only load their own direct child tags that match DDL column names. Nested repeating child nodes are handled separately in child tables.

## Implications for later `MINUS` SQLs

When we compare SRC tables to `ARGUS_APP` case tables, we should keep these rules in mind:

1. SRC sequence columns are synthetic loader keys, not Argus business keys.
2. Reconciliation should anchor first on business identifiers available in the XML, especially `SAFETYREPORTID`, case/report identifiers, product names, reactions, and other case content fields.
3. We should not expect ignored XML fields to appear in SRC unless the DDL and loader are extended.
4. Any `MINUS` SQL must account for repeating child entities separately: reactions, drugs, active substances, relatedness rows, summaries, duplicates, and linked reports.
5. Columns that are always null in the current sample set are weak candidates for initial reconciliation logic unless the Argus target mapping specifically depends on them.
